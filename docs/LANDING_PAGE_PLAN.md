# DollVerse Landing Page 方案

## 1. 页面目标

主要目标：

- 收集对产品感兴趣用户的邮箱。
- 判断“拥有玩偶、并对故事感有强需求的人”是否会被这个概念打动。
- 为后续用户访谈、共创测试、早期内测建立联系池。

次要目标：

- 让用户在很短时间内理解：这是一个让玩偶“拥有自己的世界、记忆和社交关系”的产品。
- 让用户感受到产品不是冷冰冰的 AI 硬件，而是温柔、有趣、有生命感的陪伴体验。
- 通过视觉和文案初步建立品牌气质。

## 2. 目标用户假设

当前核心用户假设：

> 拥有玩偶，且内心对“故事感”有很强需求的人。

可进一步拆解为几类早期目标用户：

1. 情感型玩偶拥有者
   - 拥有一个或多个长期陪伴自己的玩偶。
   - 会给玩偶起名字、设定性格，甚至想象它的日常生活。
   - 可能会拍摄玩偶照片、写玩偶日记、带玩偶旅行。

2. 世界观创作者
   - 喜欢搭建小世界、小房间、玩偶场景、迷你模型。
   - 对“玩偶之间也有关系、记忆和故事”有天然兴趣。
   - 可能来自手作、娃圈、棉花娃娃、BJD、IP 周边等人群。

3. 情绪陪伴需求者
   - 希望玩偶不仅是物品，而是一个可以持续互动的存在。
   - 不一定追求高频对话，更在意“它记得我”和“它像活着一样”。

4. 早期科技体验者
   - 对 AI 陪伴、智能硬件、虚拟世界感兴趣。
   - 可能愿意成为第一批测试用户，并提供反馈。

## 3. 核心定位

一句话定位：

> 给你的玩偶一颗心

用户应该理解的三件事：

- 这不是普通的智能音箱或聊天机器人。
- 这是“你的玩偶”的延伸，而不是一个新的虚拟角色。
- 你的关系、记忆和故事会成为玩偶世界的一部分。

## 4. 页面整体叙事

推荐采用“收到一封来自玩偶的信”的叙事结构。

页面不应该像传统 SaaS Landing Page 一样先讲功能、卖点和价格，而应该让用户像打开一封信一样进入体验。

推荐叙事顺序：

1. 第一屏：未打开的信封。用户点击信封区域后，通过顺滑动画切换到玩偶写来的邀请信。
2. 第二屏：展示心型硬件，说明它像玩偶的小心脏。
3. 第三屏：展示玩偶进入云端世界，与其他玩偶相遇。
4. 第四屏：邀请用户加入 WaitList。

用户情绪路径：

好奇 -> 主动打开信封 -> 被信打动 -> 理解产品形态 -> 想象自己的玩偶进入世界 -> 留下邮箱和玩偶名称。

### 5.2 色彩建议

避免：

- 过度科技感的黑蓝渐变。
- 过冷、过硬的 AI SaaS 视觉。
- 过度粉色导致产品显得只面向低龄用户。

### 5.3 字体方向

中文：

- 标题可用圆体或手写感字体。
- 正文必须优先保证可读性。
- “玩偶写信”的部分可以使用略显笨拙、圆润、手写感的字体。

英文：

- 可使用 rounded sans 或 handwriting-like font。
- 不建议使用过于潦草的 script font，因为会降低转化率。

文案风格：

- 像玩偶正在努力学习人类语言。
- 可以有轻微不完美，但不能影响理解。
- 错误要少、轻、可爱，不能像机器翻译错误。

## 6. 页面结构方案

### 6.1 顶部区域

元素：

- 左上角品牌名：DollVerse。
- 右上角语言切换按钮：中文 / English。
- 小型 WaitList 按钮，点击滚动到表单。

品牌名：

> DollVerse

命名说明：

- 中英文统一使用 DollVerse，降低品牌维护成本。
- “Doll” 直接指向玩偶，“Verse” 指向世界观、云端空间和玩偶之间的关系网络。
- 名称不绑定某一种硬件形状，后续如果硬件形态演进，不需要重命名。

### 6.2 第一屏：信封与来自玩偶的信

目标：

- 让用户立刻进入情绪。
- 用第一人称建立“玩偶正在邀请你”的感觉。
- 不要急于解释完整技术细节。
- 让用户通过一次轻量点击动作，像真的打开一封来自玩偶的信。

当前 Figma 原型理解：

- 首屏初始状态是未打开的信封，画面中有 DollVerse 品牌、语言切换、WaitList 入口、柔和背景和一个背对用户的小玩偶。
- 用户点击信封区域后，页面切换到打开后的来信状态。
- 打开后的状态中，信纸成为视觉中心，玩偶出现在信纸旁边，形成“玩偶真的把信递给你”的感觉。
- 英文和中文各有一套信封未打开与来信打开状态。

视觉：

- 初始状态是一只未打开的信封，放在柔软桌面、浅色地面或玩偶房间场景中。
- 信封旁边出现一个背对用户的玩偶，像是在等待用户打开信。
- 信封下方可以有一行很轻的提示文案，例如“Click to open a letter from your doll”或“点击打开玩偶写给你的信”。
- 点击后切换到一张柔软的信纸，像从信封中被取出并展开。
- 周围可以保留柔和的抽象色块、纸张纹理、玩偶阴影和小道具，避免画面变成普通弹窗。

交互逻辑：

1. 页面首次加载时展示未打开信封状态。
2. 用户点击信封区域。
3. 信封轻微上浮，封口打开，信纸从信封中滑出或淡入。
4. 背景和玩偶位置保持稳定，避免用户感到页面跳动。
5. 动画结束后进入玩偶来信状态，信纸成为首屏核心内容。
6. 语言切换应保留当前状态：如果用户已经打开信，再切换语言，不应回到未打开信封。

前端状态：

- `closed`：未打开信封。
- `opening`：信封打开动画进行中。
- `open`：玩偶来信已展开。

第一版不提供“重新合上信封”的入口，避免交互复杂化。

### 6.5 Waiting List 弹窗

目标：

- 明确邀请用户加入等候名单。
- 降低提交门槛。
- 提供成功反馈。

当前 Figma 原型理解：

- Waiting List 不是页面底部嵌入表单，而是点击 Join WaitList 后出现居中的 modal。
- 背景页面会被半透明遮罩压暗，用户注意力集中到表单。
- 表单字段包含 Email、Doll Name，以及一个关于用户和玩偶故事的文本区域。
- 弹窗应支持关闭按钮、提交中状态、错误状态和成功状态。

标题建议：

> 想让你的玩偶先进去看看吗？

英文：

> Want your plushie to be one of the first?

表单字段：

- Email：必填。
- 玩偶名字：必填。
- 一个你和玩偶的小故事：选填。

收集玩偶名字的目的：

- 帮助确认用户确实拥有玩偶，减少无效订阅。
- 后续邮件可以直接使用玩偶名称，让邮件更像写给用户和它的玩偶，而不是通用营销邮件。
- 让用户在提交前短暂想起自己的玩偶，提高情感投入。

关于故事字段：

- Figma 当前版本已经把故事字段放入弹窗，这有助于筛选高意向用户，也能提前获得访谈线索。
- 风险是提交门槛升高，尤其移动端会让表单显得更重。
- 推荐策略：MVP 默认邮箱和玩偶名称必填，故事字段选填；如果希望更强筛选，可以单独测试“故事字段必填”的版本。

成功提示中文：

> 订阅成功。  
> 小信箱已经亮起来了。等世界准备好，我们会写信给你和你的玩偶。

成功提示英文：

> You are on the list.  
> The little mailbox is glowing now. We will write to you and your plushie when the world is ready.

错误提示：

- 邮箱为空：请输入你的邮箱。
- 邮箱格式错误：这个邮箱看起来还没有写好。
- 玩偶名字为空：请告诉我们你的玩偶叫什么。
- 重复提交：你已经在名单里了。小信箱还记得你。
- 网络错误：信好像没有送出去，请再试一次。

## 10. 实现方案

### 10.1 推荐技术路线

第一版推荐使用轻量前端 + Serverless 后端。

前端：

- Next.js 或 Astro。
- 如果只做单页静态站，Astro 更轻。
- 如果未来可能扩展登录、用户故事、后台管理，Next.js 更合适。

推荐：

> Next.js + Vercel + Supabase

原因：

- 部署快。
- 表单 API 容易实现。
- Supabase 可以直接存储邮箱、玩偶名称、时间、国家、来源参数。
- 后续可以扩展用户访谈标签、内测状态、邮件发送状态。

### 10.2 前端模块

页面组件：

- `Header`
- `LanguageToggle`
- `EnvelopeHero`
- `EnvelopeClosed`
- `LetterOpen`
- `HardwareShowcase`
- `CloudWorld`
- `WaitlistModal`
- `Footer`

状态：

- `language`
- `letterState`
- `isWaitlistOpen`
- `email`
- `plushieName`
- `plushieStory`
- `submitStatus`
- `errorMessage`
- `isSubmitting`

表单校验：

- 前端先做基本 email 格式校验。
- 后端再次校验，不能只依赖前端。

### 10.3 后端接口

接口：

`POST /api/waitlist`

请求体：

```json
{
  "email": "user@example.com",
  "language": "zh",
  "plushieName": "Momo",
  "plushieStory": "Momo sits by my bed and watches rain with me.",
  "source": "landing_page"
}
```

后端处理：

1. 标准化邮箱：trim + lowercase。
2. 校验邮箱格式。
3. 校验玩偶名称：trim 后不能为空，限制长度，例如 1-40 个字符。
4. 如果提交了玩偶故事，限制长度，例如 0-500 个字符。
5. 读取国家信息，例如从 Vercel/Cloudflare 提供的国家 header 中读取，只保存国家，不保存 IP。
6. 读取 User-Agent。
7. 读取 UTM 参数或 referrer。
8. 写入数据库。
9. 如果邮箱已存在，返回友好的重复提示。

返回示例：

```json
{
  "ok": true,
  "status": "subscribed"
}
```

错误示例：

```json
{
  "ok": false,
  "error": "invalid_email"
}
```

### 10.4 数据库设计

表名：

`waitlist_subscribers`

字段建议：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | uuid | 主键 |
| email | text | 用户邮箱，唯一 |
| language | text | zh / en |
| plushie_name | text | 玩偶名称，必填 |
| plushie_story | text | 用户和玩偶的小故事，选填 |
| country | text | 国家或地区代码，例如 US / CN / JP |
| user_agent | text | 浏览器信息 |
| referrer | text | 来源页面 |
| utm_source | text | UTM source |
| utm_medium | text | UTM medium |
| utm_campaign | text | UTM campaign |
| created_at | timestamptz | 订阅时间 |

约束：

- `email` unique。
- `plushie_name` not null。
- `created_at` 默认当前时间。

### 10.5 国家记录与隐私

当前策略：

- 不保存访问者 IP。
- 只保存国家或地区代码，用于理解早期用户的大致分布。
- 国家信息优先由部署平台或 CDN 在请求处理中提供，例如 Vercel/Cloudflare 的地理位置 header。
- 后端只把国家代码写入数据库，不把原始 IP 写入数据库。

建议：

- 页面底部增加一句简短说明：

> We collect your email, your plushie's name, optional story, and country-level visit information to manage the waitlist and understand early interest.

中文：

> 我们会记录你的邮箱、玩偶名称、你选填的小故事和国家级别的访问信息，用于管理等候名单和了解早期用户分布。

关于是否需要隐私协议：

- 即使不记录 IP，只要收集邮箱，就已经在处理个人信息，因此不建议完全没有隐私说明。
- 第一版可以先放一段简短、清晰的隐私说明，并链接到一个轻量 Privacy Policy 页面。
- 如果页面会投放到欧盟、英国、加州等地区，或后续接入邮件营销、数据分析工具，建议准备正式 Privacy Policy。

第一版可执行策略：

- 保存 email、plushie_name、plushie_story、created_at、language、country、referrer、utm。
- 不保存完整 IP。
- 不把玩偶名称公开展示或分享给第三方。
- 邮件中可使用玩偶名称做个性化称呼，例如“DollVerse 给 Momo 的第一封信”。

### 10.6 防滥用策略

第一版最低限度：

- 前端禁用重复提交。
- 后端根据 email unique 去重。
- 使用平台或 CDN 提供的基础频率限制，不把 IP 写入业务数据库。
- 增加 honeypot 隐藏字段，减少垃圾提交。

如果上线后有垃圾提交：

- 增加 Cloudflare Turnstile。
- 增加邮件确认流程。

## 11. 动画与交互建议

推荐轻量动画：

- 首屏信封打开：用户点击信封区域后，信封封口打开，信纸滑出或淡入，玩偶来信状态接管画面。
- 心型硬件呼吸灯：CSS animation。
- 云朵缓慢漂浮：CSS transform。
- 信纸入场：轻微上浮和淡入。
- 成功提交后：小信箱亮起或信封飞入。

### 11.1 首屏信封动画实现建议

推荐方案：

> Next.js + Framer Motion + CSS layered assets

原因：

- 这个动画是 UI 状态转换，不是复杂角色动画，用 Framer Motion 管理 `closed -> opening -> open` 更合适。
- Framer Motion 支持 `AnimatePresence`、layout animation、opacity/scale/translate/rotate 的组合，能比较自然地完成信封、信纸、玩偶和背景之间的顺滑过渡。
- 动画仍然可以保持轻量，不需要引入大型 3D 或游戏引擎。

实现方式：

1. 将首屏拆成几个可独立动画的图层：背景色块、玩偶、信封背板、信封封口、火漆印、信纸。
2. 初始状态只显示信封，信纸隐藏或缩在信封后方。
3. 点击信封后进入 `opening` 状态：
   - 信封整体轻微上浮，例如 `y: -8`。
   - 封口轻微旋转打开，例如 `rotateX` 或 2D `rotate`。
   - 火漆印淡出或缩小。
   - 信纸从信封后方上滑并放大到最终位置。
4. 动画完成后进入 `open` 状态：
   - 信封淡出或退到背景。
   - 信纸成为主内容。
   - 玩偶从侧边轻微进入或保持在信纸旁边。
5. 语言切换只替换信纸文案，不重置 `letterState`。

工具取舍：

- CSS transition：适合呼吸灯、云朵漂浮、按钮 hover 等简单循环或微交互。
- Framer Motion：适合首屏信封打开、modal 入场、信纸状态切换这类有状态的 UI 动画。
- Lottie：只有在设计师希望制作复杂、手绘感很强的信封 opening 动画，并能提供 After Effects/Lottie 文件时使用。
- Rive：适合更复杂的交互式角色动画，例如玩偶眨眼、挥手、跟随鼠标等；第一版不必使用。
- GSAP：能力足够强，但对当前 MVP 来说偏重，除非后续需要复杂时间轴和多段滚动动画。

第一版建议：

- 信封动画用 Framer Motion 实现。
- 背景漂浮、硬件呼吸灯、按钮 hover 用 CSS 实现。
- 等待名单 modal 入场也用 Framer Motion，保持动画体系统一。

避免：

- 大量复杂滚动动画。
- 首屏加载很慢的 3D 场景。
- 会遮挡表单的装饰元素。

性能目标：

- 首屏 2 秒内可读。
- 移动端优先适配。
- 图片使用 WebP/AVIF。
- 动画不影响低性能设备操作。

## 13. 转化测试假设

第一版可以重点验证：

1. 信件叙事是否比传统产品介绍更能驱动订阅。
2. “藏在衣服里的心型硬件”是否足够有记忆点。
3. 用户更被“陪伴记忆”打动，还是更被“玩偶社交世界”打动。
4. 中英文用户对“轻微语法错误”的接受度是否不同。

建议后续 A/B 测试：

- Hero 文案版本 A：玩偶写信。
- Hero 文案版本 B：直接讲产品价值。
- CTA 文案 A：加入等候名单。
- CTA 文案 B：让我的玩偶先进去。
- 表单字段 A：邮箱 + 玩偶名字。
- 表单字段 B：邮箱 + 玩偶名字 + 一个可选故事问题。

## 14. 第一版交付范围

建议 MVP Landing Page 包含：

- 单页响应式页面。
- 中英文切换。
- 玩偶邀请信。
- 心型硬件展示区域。
- 云端玩偶世界展示区域。
- 邮箱 Waiting List 表单。
- 前后端邮箱格式校验。
- 订阅成功/失败提示。
- 数据库存储邮箱、玩偶名称、时间、语言、来源、国家。

不建议第一版包含：

- 登录系统。
- 真实玩偶世界交互。
- 复杂后台管理。
- 大量角色设定。
- 价格页。
- 完整产品规格页。

## 16. 待确认问题

后续制作页面前建议确认：

- DollVerse 是否需要搭配中文副标题，例如“玩偶世界”或“给玩偶的云端世界”。
- 心型硬件优先采用缝制方案、磁吸方案，还是两者都支持。
- 产品第一批目标市场是中文用户、英文用户，还是同时测试。
- 是否需要在订阅成功后追问一个可选故事问题，例如“你希望 DollVerse 记住你和玩偶的哪件小事？”
- 是否有现成 Logo、硬件草图或玩偶形象。
- 是否需要接入邮件服务，订阅成功后自动发送欢迎邮件。

## 18. 总结

这个 Landing Page 的关键不是把功能讲全，而是让用户产生一个清晰想象：

> 我的玩偶不是静静坐在那里。它也许有自己的世界，也记得我。

因此首版应该优先做好三件事：

- 用“玩偶写信”建立情感入口。
- 用“藏在衣服里的心型硬件”建立产品记忆点。
- 用“云端玩偶世界”建立未来想象。

Waiting List 的表单必须简单、清晰、可信。用户在被故事打动后，应该能用邮箱和玩偶名称轻松完成表达兴趣的动作；故事字段可以作为选填信息或转化测试项。

## 19. Figma UI 设计 Prompt

可直接复制给 Figma AI、设计师或其他 UI 生成工具：

```text
请为 DollVerse 设计一个高保真响应式 Landing Page UI，包含桌面端和移动端版本。

项目背景：
DollVerse 是一个面向拥有玩偶、并对故事感有强需求用户的产品。产品通过一个心型智能硬件，让用户的玩偶携带与主人之间的关系记忆，进入云端玩偶世界，与其他玩偶互动、写信、交换故事，并把故事带回给主人。Landing Page 的核心目标是收集 Waiting List 用户的邮箱、玩偶名称，以及可选的用户和玩偶故事。

品牌与情绪：
- 品牌名固定为 DollVerse，中英文都使用 DollVerse。
- 整体氛围应友好、温暖、有趣、柔软、像一个安全的小世界。
- 视觉灵感可以接近 Animal Crossing 带来的 cozy、playful、tiny world 感受，但不能复制任何具体角色、UI、Logo 或可识别版权元素。
- 页面应该像用户收到一封玩偶写来的邀请信，而不是传统 SaaS 产品页。
- 设计要有故事感、手作感和轻微童话感，但不要幼稚，也不要像儿童教育产品。

页面结构：
1. Header
   - 左侧显示 DollVerse 品牌名。
   - 右侧包含语言切换按钮：中文 / English。
   - 右侧可包含一个轻量 CTA：Join Waitlist / 加入等候名单。

2. Hero Letter
   - 首屏初始状态不是直接展示信，而是展示一只未打开的信封。
   - 用户点击信封区域后，画面通过顺滑小动画切换到玩偶来信状态。
   - 动画应表现信封被打开、信纸滑出或淡入，最后信纸成为视觉中心。
   - 信纸内容是一封由玩偶写给用户的信，主题是“邀请你加入我的世界”。
   - 信纸应像放在柔软桌面、草地或玩偶房间中的一张信。
   - 周围可有纽扣、线团、小花、贴纸、柔和灯点、信封等小元素。
   - 初始状态可让玩偶背对用户或坐在信封旁边；打开状态可让玩偶出现在信纸旁边。
   - Hero 中需要有主 CTA：让我的玩偶先进去 / Let my plushie in first。
   - 次 CTA：看看它的世界 / See its little world。

3. Heart Hardware Section
   - 展示一个心型智能硬件。
   - 硬件不是外露挂件，而是被缝制或磁吸在玩偶衣服内。
   - 它像玩偶藏在衣服里的小心脏。
   - 硬件风格应简洁、圆润、精致，有 Apple-like industrial design 的感觉，但仍然温暖。
   - 需要表现呼吸灯、小巧type-C充电口、两个低存在感按钮。
   - 请设计一个安装示意：缝制在衣服内，或通过磁吸固定在衣服内侧。

4. Cloud Plushie World Section
   - 展示玩偶进入云端世界后的场景。
   - 场景可以是云端小岛、草地村庄、小路、发光邮箱、树屋、柔软云朵。
   - 出现几只不同玩偶，例如泰迪熊、小象、猴子等，但都应是原创形象。
   - 用户自己的玩偶站在入口处，衣服里藏着心型硬件，胸前或口袋边缘透出柔和灯光。
   - 重点传达：玩偶带着你们的记忆，去认识别的玩偶。

5. Waiting List Modal
   - 点击 Join Waitlist / 加入等候名单后，打开居中的 modal，而不是跳转到新页面。
   - 背景页面需要用半透明遮罩压暗。
   - 表单字段：Email，Plushie name / 玩偶名字，A story about you and your doll / 你和玩偶的小故事。
   - Email 和玩偶名字必填。
   - 故事字段建议作为选填，除非要测试高门槛筛选版本。
   - CTA 文案：加入等候名单 / Join the waitlist。
   - 成功状态需要温暖、有故事感，例如小信箱亮起，或信封飞入邮箱。
   - 错误状态需要清晰但不冷冰冰。
   - 表单旁边可以显示简短说明：我们会记录你的邮箱、玩偶名称、你选填的小故事和国家级别的访问信息，用于管理等候名单和了解早期用户分布。

视觉风格：
- 避免黑蓝科技感、过强 AI SaaS 感、过度粉色、过度低龄化。
- 页面应明亮、柔和、有空气感。
- 可以使用柔和阴影、纸张纹理、布料纹理、手作边缘、轻微拟物细节。
- 卡片圆角控制在 8px 左右，不要大量使用大圆角卡片嵌套。
- 不要使用深色模糊背景、抽象渐变球、装饰性 bokeh。

字体与排版：
- 标题使用圆润、友好的字体。
- 玩偶信件部分可以使用略显笨拙但清晰可读的手写感字体。
- 正文必须易读，尤其移动端。
- 英文文案可以保留少量轻微语法不完美，表达玩偶正在努力学习人类语言，但不要影响理解。

交互与状态：
- 设计语言切换状态。
- 设计首屏信封未打开状态、信封打开动画中的中间状态、玩偶来信打开状态。
- 设计表单默认、输入中、错误、提交中、成功状态。
- 设计心型硬件呼吸灯的视觉状态。
- 设计轻量动画建议，例如信封打开、信纸滑出或淡入、云朵缓慢漂浮、小信箱成功后亮起。
- 首屏信封动画建议后续用 Framer Motion 实现，因此设计应尽量拆分可动画图层：背景、玩偶、信封背板、信封封口、火漆印、信纸。

响应式要求：
- 输出桌面端 1440px 宽设计。
- 输出移动端 390px 宽设计。
- 移动端第一屏必须仍然能看到 DollVerse、玩偶来信、主 CTA，并露出一点下一段内容。
- 移动端未打开信封状态和打开后信纸状态都要单独设计。
- 所有按钮和输入框在移动端必须足够大，文本不能溢出或遮挡。

不要做：
- 不要做传统 B2B SaaS 风格。
- 不要做价格页、功能表格或复杂导航。
- 不要复制 Animal Crossing 的具体视觉资产。
- 不要把硬件画成草莓，它必须是心型。
- 不要把硬件设计成外露吊坠，重点是缝制或磁吸在玩偶衣服内。

最终交付：
- 一个完整 Landing Page 高保真 UI。
- 桌面端与移动端布局。
- 关键组件状态：语言切换、表单校验、提交成功。
- 简单设计系统：颜色、字体、按钮、输入框、CTA、信纸组件、硬件展示组件。
```
